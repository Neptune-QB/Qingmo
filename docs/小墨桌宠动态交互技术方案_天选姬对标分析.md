# 小墨桌宠动态与交互 — 技术方案（对标华硕天选姬）

> 基于天选姬技术方案对标分析，制定小墨 V1.0（Compse 原生动画增强）+ V2.0（Live2D 深度集成）两阶段路线

---

## 一、华硕天选姬技术方案全解析

天选姬是华硕天选系列笔记本预装的系统级桌面精灵，经过多年迭代，其技术栈分两条路径：

### 方案 A：Live2D Cubism SDK（Android 端）

| 维度 | 技术细节 |
|---|---|
| **动画引擎** | Live2D Cubism SDK for Native（C++ 核心，Android NDK 编译） |
| **渲染层** | OpenGL ES 2.0/3.0，通过 `GLSurfaceView` 或 `TextureView` 渲染 |
| **模型格式** | `.moc3`（Live2D 模型文件）+ `.physics3.json`（物理参数）+ `.cdi3.json`（部件索引） |
| **物理引擎** | Cubism 内置物理引擎：头发/衣摆/挂饰随重力、移动惯性自然摆动 |
| **眼球追踪** | 逆运动学参数混合：瞳孔始终跟随触控点/光标，实现"注视"效果 |
| **碰撞检测** | 身体各部位（头/胸/手）绑定碰撞体，点击不同部位触发不同动画组 |
| **网格变形** | 支持拖拽拉伸四肢/头部，Mesh 实时形变回弹 |
| **窗口管理** | `WindowManager.addView()` + `TYPE_APPLICATION_OVERLAY`，系统级悬浮窗 |
| **系统感知** | 监听电量、CPU 温度、时间，切换对应 motion group 和表情参数 |
| **交互反馈** | 点击 → 音效 + 对话气泡 + 表情切换 + 粒子特效 |

**Android 集成架构（Live2D 路线）**：
```
┌─────────────────────────────────────┐
│  WindowManager (TYPE_APPLICATION_OVERLAY) │
│  ┌─────────────────────────────────┐ │
│  │  GLSurfaceView                  │ │
│  │  ┌───────────────────────────┐  │ │
│  │  │ Live2D Cubism Core (C++) │  │ │
│  │  │  ├─ Model (.moc3)         │  │ │
│  │  │  ├─ Physics (.json)       │  │ │
│  │  │  ├─ Motion (.json)        │  │ │
│  │  │  └─ Parameter Blending    │  │ │
│  │  └───────────────────────────┘  │ │
│  └─────────────────────────────────┘ │
│  Kotlin Bridge Layer — 状态控制/事件分发 │
└─────────────────────────────────────┘
```

### 方案 B：Unity + MMD（Windows 桌面端）

| 维度 | 技术细节 |
|---|---|
| **引擎** | Unity 3D |
| **模型来源** | 官方 MMD 模型（PMX 格式） |
| **模型转换** | MMD4Mecanim 插件：PMX → FBX |
| **动作数据** | 官方 MMD VMD 动作文件 → Unity AnimationClip |
| **动作控制** | Unity Animator Controller（状态机 + Blend Tree） |
| **窗口透明** | `SetWindowLong` + `WS_EX_LAYERED` + `SetLayeredWindowAttributes` |
| **窗口置顶** | `SetWindowPos(HWND_TOPMOST, ...)` |
| **点击穿透** | `WS_EX_TRANSPARENT` 扩展样式 |

> ⚠️ Unity 方案是 **Windows 桌面端** 专属方案，不适合 Android 端小墨。

## 二、当前小墨 vs 天选姬 — 架构差距

| 维度 | 当前小墨 | 天选姬 |
|---|---|---|
| **动画引擎** | Compose `animateFloat`（位移） | Live2D Cubism SDK（骨骼+网格+物理） |
| **形象资源** | 1 张静态 PNG | 分层 PSD → Live2D 模型（数十层部件） |
| **动态层级** | L1：位置浮动（±3dp） | L1+L2+L3：位置移动 + 姿态切换 + 物理摆动 |
| **渲染方式** | Compose `Image` | OpenGL ES 原生渲染 |
| **显示范围** | 播放器页面内嵌 | 系统级悬浮窗（任意 App 之上） |
| **眼球追踪** | ❌ 无 | ✅ 瞳孔跟随触控点 |
| **碰撞检测** | ❌ 无 | ✅ 身体部位绑定碰撞体 |
| **系统感知** | ❌ 无 | ✅ 电量/温度/时间驱动行为 |
| **入口** | 进入播放器 Page | 系统启动后常驻 |

## 三、推荐方案：两阶段演进路线

### 结论：V1.0 不引入 Live2D，纯 Compose 动画增强；V2.0 引入 Live2D Cubism SDK

#### 为什么 V1.0 不用 Live2D？

1. **时间成本**：Live2D 模型制作需美术切分 30+ 图层部件 + 绑定骨骼参数，至少 2 周；当前仅 1 张 PNG
2. **集成复杂度**：NDK 编译 + JNI 桥接 + GLSurfaceView 嵌入 Compose 的 `AndroidView`，调试周期约 1 周
3. **比赛节点**：6 月 11 日交付，剩余 13 天，必须保证核心功能可演示
4. **PRD 明确**：V1.0 简方案（Canvas + Emoji），后续迭代再升级

#### V2.0 Live2D 的不可替代价值

- **物理级真实感**：头发/衣摆/挂饰随拖拽惯性自然摆动，PNG 动画无法模拟
- **眼球追踪**：小墨的注视点始终跟随用户手指，交互沉浸感质的飞跃
- **Mesh 形变**：戳头/拽手产生弹性形变反馈，桌宠「生命感」关键
- **参数化表情**：0-1 滑条控制喜怒哀乐，而非切换整张图片，过渡更自然

## 四、V1.0 方案：Compose 原生动画增强（比赛前交付）

### 4.1 三层动态体系

基于你之前定义的桌宠三层动态模型，用 Compose 原生 API 完整实现：

```
┌─────────────────────────────────────────────────┐
│ L1 位置移动层                                     │
│   animateFloatAsState → 弹性拖拽跟随               │
│   Animatable → 点击目标位置缓动                     │
│   spring(dampingRatio=0.3f) → 松手回弹             │
├─────────────────────────────────────────────────┤
│ L2 姿态切换层                                     │
│   Crossfade → 闲置/点击/拖拽 三态切换               │
│   AnimatedContent → 同屏多姿态共享元素过渡           │
│   graphicsLayer { scaleX/scaleY } → 按压缩放        │
├─────────────────────────────────────────────────┤
│ L3 表情反馈层                                     │
│   Canvas → 粒子特效（爱心爆炸/星光粒子）             │
│   AnimatedVisibility → 对话气泡弹出/消散            │
│   animateFloatAsState → 表情渐变切换（alpha 交叉）   │
└─────────────────────────────────────────────────┘
```

### 4.3 弹性拖拽实现

```kotlin
// 拖拽状态
val offsetX = remember { Animatable(0f) }
val offsetY = remember { Animatable(0f) }

// 拖拽手势
Modifier.pointerInput(Unit) {
    detectDragGestures(
        onDragEnd = {
            // 松手回弹到原位
            launch {
                offsetX.animateTo(0f, spring(dampingRatio = 0.3f))
                offsetY.animateTo(0f, spring(dampingRatio = 0.3f))
            }
        }
    ) { change, dragAmount ->
        change.consume()
        launch {
            offsetX.snapTo(offsetX.value + dragAmount.x)
            offsetY.snapTo(offsetY.value + dragAmount.y)
        }
    }
}

// 弹性跟随：小墨偏移量与手指位移 1:0.6（惯性感）
offsetX.snapTo(offsetX.value + dragAmount.x * 0.6f)
```

### 4.5 碰撞检测模拟（无需真实 3D 碰撞体）

```kotlin
// 将 PNG 划分为简单矩形热区
enum class BodyPart(val bounds: Rect) {
    Head(Rect(0.2f, 0f, 0.8f, 0.4f)),       // 头部 20%-80% 宽度，0-40% 高度
    Body(Rect(0.1f, 0.4f, 0.9f, 0.7f)),      // 身体
    HandLeft(Rect(0f, 0.3f, 0.2f, 0.6f)),   // 左手
    HandRight(Rect(0.8f, 0.3f, 1f, 0.6f)),  // 右手
}

fun detectBodyPart(tapOffset: Offset, viewSize: Size): BodyPart {
    val normalized = Offset(tapOffset.x / viewSize.width, tapOffset.y / viewSize.height)
    return BodyPart.entries.firstOrNull { it.bounds.contains(normalized) }
        ?: BodyPart.Body // 默认命中身体
}
```

### 4.7 粒子特效（Canvas 纯代码）

```kotlin
@Composable
fun ParticleEffect(
    type: ParticleType, // Hearts, Stars, Confetti
    trigger: Boolean,
) {
    val particles = remember { mutableStateListOf<Particle>() }
    
    LaunchedEffect(trigger) {
        if (trigger) {
            // 生成 20 个粒子
            repeat(20) {
                particles.add(Particle(
                    x = Random.nextFloat(),
                    y = 1f,
                    velocityX = Random.nextFloat(-0.02f, 0.02f),
                    velocityY = Random.nextFloat(-0.03f, -0.01f),
                    alpha = 1f,
                    scale = Random.nextFloat(0.5f, 1.5f),
                ))
            }
            // 粒子消散
            delay(2000)
            particles.clear()
        }
    }
    
    Canvas(modifier = Modifier.fillMaxSize()) {
        particles.forEach { particle ->
            drawCircle(
                color = when (type) {
                    ParticleType.Hearts -> Color(0xFFFF6B8A) // 粉色爱心
                    ParticleType.Stars -> Color(0xFFFFD700)  // 金色星星
                    ParticleType.Confetti -> Color(0xFF5B8C85) // 青白色碎片
                }.copy(alpha = particle.alpha),
                radius = 6.dp.toPx() * particle.scale,
                center = Offset(
                    particle.x * size.width,
                    particle.y * size.height,
                ),
            )
        }
    }
}
```

### 5.2 Live2D 模型制作流程

```
原始设计稿（xiaomo.png）
    ↓ 美术在 Photoshop / Live2D Cubism Editor 中切分图层
分层 PSD（头/身体/手臂×2/腿×2/头发×4/衣摆×3/挂饰×2 ~ 30+ 层）
    ↓ Live2D Cubism Editor 绑定骨骼 + 参数 + 物理
Live2D 模型文件：
  ├─ xiaomo.moc3          — 模型网格数据
  ├─ xiaomo.model3.json   — 模型配置文件
  ├─ xiaomo.physics3.json — 物理参数（头发/衣摆摆动）
  ├─ motions/             — 动作文件（idle/click/drag/poke）
  │   ├─ idle.motion3.json
  │   ├─ poke_head.motion3.json
  │   └─ drag.motion3.json
  └─ textures/            — 贴图文件
      └─ xiaomo.1024.png
```

### 5.3 Android 集成架构

```kotlin
// 1. 添加依赖
// build.gradle.kts
dependencies {
    implementation("com.live2d.cubism:cubism-sdk-android:5.0.0")
}

// 2. Live2D 渲染器封装为 Compose 组件
@Composable
fun Live2DXiaoMoView(
    modelPath: String = "models/xiaomo.model3.json",
    motionState: Live2DMotion,
    tapCallback: (String) -> Unit,  // hitArea 回调
    modifier: Modifier = Modifier,
) {
    AndroidView(
        factory = { context ->
            GLSurfaceView(context).apply {
                setEGLContextClientVersion(2)
                setRenderer(Live2DRenderer(modelPath))
                setOnTouchListener { _, event ->
                    // 射线检测碰撞
                    val hitArea = renderer.hitTest(event.x, event.y)
                    if (hitArea != null) tapCallback(hitArea)
                    true
                }
            }
        },
        modifier = modifier
    )
}

// 3. 窗口悬浮（系统级）
class XiaoMoOverlayService : Service() {
    private lateinit var windowManager: WindowManager
    
    override fun onCreate() {
        val layoutFlag = if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O)
            WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY
        else
            WindowManager.LayoutParams.TYPE_PHONE

        val params = WindowManager.LayoutParams(
            300, 400, layoutFlag,
            WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE,
            PixelFormat.TRANSLUCENT
        )
        windowManager.addView(live2DView, params)
    }
}
```

## 六、方案对比总结

| | V1.0 Compose 原生增强 | V2.0 Live2D Cubism |
|---|---|---|
| **开发周期** | 6 天 | 3-4 周（含模型制作） |
| **美术成本** | 3 张 PNG 切图 | 30+ 图层 PSD + 骨骼绑定 |
| **包体积增量** | 0 KB（纯代码） | ~3 MB（SDK + 模型文件） |
| **性能开销** | 极低（Compose 硬件加速） | 中等（OpenGL 独立渲染线程） |
| **安全感** | ✅ 6月4日前可交付 | ❌ 比赛前不可交付 |
| **动态真实感** | ★★★☆☆ | ★★★★★ |
| **扩展性** | 姿态/特效靠新增代码 | 模型参数即素材 |
| **技术风险** | 低（现有 Compose 体系） | 中（NDK 兼容性 + JNI 稳定性） |

### 最终建议

**比赛前（6月4日）：执行 V1.0 方案**，用 Compose 原生动画实现弹性拖拽 + 姿态切换 + 表情反馈 + 粒子特效，在现有 1 张 PNG + 追加 3 张切图基础上完成「桌宠感」首版。

**比赛后（6月12日起）：启动 V2.0 评估**，根据用户反馈和产品方向决定是否投入 Live2D 深度集成。如决定做，需安排 2 周美术 + 2 周引擎集成的完整周期。
