## 难点5：SQL关键词检索对口语化查询完全失效

RAG 的核心价值是"让 LLM 基于真实剧情回答"。但第一版实现直接用 SQL LIKE 做关键词匹配——这个方案在处理用户自然口语时几乎 100% 失效。

### 坑一：关键词提取只是粗暴拆词，不懂语义

用户提问 "刚从监狱出来的那个男人后来怎么样了"，关键词提取逻辑去掉 "的""那个""后来""怎么样了" 等停用词后，得到 `["监狱", "男人"]`。数据库里存的剧情知识是：

```
drama_characters: "项云峰因盗墓入狱7年，出狱后遭债主逼债..."
drama_timeline:   "@120s [冲突] 债主上门讨债"
```

`LIKE '%监狱%'` 和 `LIKE '%男人%'` 在这两段文本中均为 **零命中**——数据库用的是 "入狱""出狱"，用户说的是 "监狱"；数据库用的是 "项云峰"，用户说的是 "男人"。字面零交集，关键词匹配颗粒无收。

这不是个例。以下真实用户提问在 LIKE 通道中全部命中率为 0：

| 用户口语 | 数据库原文 | 为什么失效 |
|----------|-----------|-----------|
| "刚从监狱出来的男人" | "项云峰因盗墓入狱7年" | 监狱≠入狱，男人≠项云峰 |
| "那个女的是谁" | "李静（女主）：古董鉴定师" | 女的≠女主，谁≠李静 |
| "讲了什么故事" | "项云峰出狱遭遇逼迫..." | 故事≠1000字摘要中的任何一个词 |

### 坑二：结果为空时 LLM 被迫编造

LIKE 返回空 → RAG context 为空 → LLM system prompt 中虽然写了"必须基于本地真实数据回答"，但没有数据就只能拒绝回答或凭模型内部知识编造。用户在对话中收到的回复是 "关于这个问题，我目前还没有相关数据"——Agent 在知识库明明有信息的情况下说自己不知道，体验彻底崩坏。

### 解决方案（向量语义检索替代关键词匹配）

**① 文本 → 向量编码 (BGE-small-zh-v1.5)**

将全部剧情知识（角色、时间线、高光点、每集摘要）逐条转换为 512 维语义向量。BGE 模型专门针对中文语义优化，"监狱出来"和"入狱出狱"虽然在字面上不同，但在 512 维空间里被编码为高度相似的向量——余弦相似度 0.53，远高于无关文本的 0.12。

| 查询 | 最高命中 | 相似度 | 来源 |
|------|---------|--------|------|
| "刚从监狱出来的男人" | 项云峰（男主）：盗墓世家传人，入狱7年... | 0.53 | characters |
| "那个女的是谁" | 李静（女主）：古董鉴定师 | 0.61 | characters |
| "讲了什么故事" | 项云峰出狱遭遇逼迫，债主上门讨债... | 0.48 | summary |

**② FAISS 向量索引 (IndexFlatIP + L2 归一化)**

114 条知识文档 → 512 维向量 → FAISS 内存索引，检索耗时 < 1ms。`faiss.normalize_L2()` 将所有向量模长归一化到 1，此时内积 = 余弦相似度，直接取 top-5 最高分。

**③ 结果去重注入 LLM context**

向量返回的结果与戏剧名/角色名等精确匹配字段合并，按 40 字符前缀去重后取前 5 条，作为"强制参考知识"注入 system prompt：

```
⚠️ 你必须完全基于以下本地真实数据库已存储的剧情内容回答问题。
如以下知识无法覆盖用户的问题，请如实说明「数据库中暂无相关信息」。
请勿使用你自己的常识或训练数据中的剧情知识进行回答。

【语义关联知识】
1. [角色] 项云峰（男主）：盗墓世家传人。因参与盗墓活动被判处七年有期徒刑...
2. [事件] @120s [冲突] 债主上门讨债，威胁项云峰还钱...
```

**④ 离线构建，不阻塞服务启动**

BGE 模型约 95MB，首次下载需访问 huggingface.co。本机网络对该域名存在 SSL 阻断，启动时同步下载会阻塞 FastAPI 10 分钟。解决方案：

- `huggingface-cli download` 命令行预缓存模型到 `~/.cache`
- 启动设 `HF_HUB_OFFLINE=1` 跳过网络检查
- 向量索引构建放在后台线程异步执行，不阻塞 API 服务
- 构建期间 `vector_rag.ready = False` → Agent 自动跳过向量通道，不影响 AI 对话可用性

### 效果

50 条测试问题对比：

| 指标 | 纯 LIKE | 向量 RAG |
|------|---------|----------|
| 口语化查询命中 | 0% | 100% |
| 整体准确率 | 19.9% | 41.9% |
| 检索耗时 | ~3ms (6 条 SQL) | < 1ms (114 条向量) |

准确率翻倍但绝对值仍偏低（41.9%）——根因不在检索，在于知识库仅 114 条文档。随着视频分析流水线逐集填充，数据量增长会自动提升覆盖率。

### 小结

SQL LIKE 匹配的是字形，向量匹配的是语义。中文口语化查询的多样性决定了纯关键词方案的天花板极低——用户永远会换着花样问同一个意思。向量检索不是锦上添花，是从"偶尔能答"到"基本能答"的质变。
